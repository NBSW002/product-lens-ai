# ProductLens 手动测试文档

## 1. 测试目标

本文档用于指导人工测试 ProductLens 系统，覆盖正常流程、错误流程、边界条件、极端情况和部署场景。

手动测试重点：

- 页面是否能正常访问。
- Amazon 链接是否能正确创建任务。
- 分析流水线是否完整展示。
- 商品抓取、图片分析、文本分析、质量检查、自动修订是否按预期执行。
- 失败时是否能看到明确错误。
- 重试是否可用。
- 质量检查是否能拦截无依据卖点。
- 部署到云服务器后 Nginx、后端服务、API 代理是否正常。

## 2. 测试环境

### 2.1 本地测试环境

前端：

```bash
cd frontend
npm install
npm run dev
```

后端：

```bash
cd backend
python -m uvicorn app.main:app --reload --port 8000 --env-file .env
```

访问：

```text
http://127.0.0.1:5173
```

### 2.2 云服务器测试环境

推荐路径：

```text
/opt/product-lens-ai
```

前端由 Nginx 托管：

```text
/opt/product-lens-ai/frontend/dist
```

后端由 systemd 管理：

```text
product-lens-backend.service
```

访问：

```text
http://服务器公网 IP
```

## 4. 基础可用性测试

### TC-001 首页访问

测试步骤：

1. 打开浏览器。
2. 访问系统首页。

预期结果：

- 页面能正常打开。
- 顶部显示 ProductLens。
- 页面有 Amazon 商品链接输入框。
- 页面显示当前模式：Demo、真实 API 或服务未连接。

异常判断：

- 如果出现 Nginx 500，检查 `/var/log/nginx/error.log`。
- 如果出现空白页，检查浏览器 Console 和前端构建产物。

### TC-002 健康检查接口

测试步骤：

```bash
curl -i http://127.0.0.1:8000/api/health
```

预期结果：

- 返回 HTTP 200。
- JSON 中包含 `status` 和 `mode`。

示例：

```json
{"status":"ok","mode":"live"}
```

### TC-003 前端到后端代理

测试步骤：

1. 打开前端页面。
2. 查看页面右上角模式。

预期结果：

- 如果后端可用，显示 Demo 或真实 API。
- 如果后端不可用，显示服务未连接。

## 5. 正常流程测试

### TC-101 Demo 模式完整分析

前置条件：

- `APP_MODE=demo`
- 后端已启动。
- 前端已启动。

测试步骤：

1. 打开页面。
2. 点击“使用示例商品”。
3. 点击“开始分析”。
4. 等待任务完成。

预期结果：

- 任务进度从校验、抓取、图片分析、文本分析、质量检查逐步推进。
- 页面展示商品摘要。
- 页面展示目标用户、使用场景、痛点、卖点。
- 页面展示短视频口播。
- 页面展示质量报告。
- 分析过程区域可以展开。
- FINALIZE 阶段为 completed。

### TC-102 Live 模式真实商品分析

前置条件：

- `APP_MODE=live`
- 三个 API key 已配置。
- 后端已重启。

测试步骤：

1. 输入有效 Amazon 商品链接。
2. 点击开始分析。
3. 等待分析完成。
4. 展开每个流水线阶段。

预期结果：

- PRODUCT_FETCH 输出商品标题、价格、评分、图片等信息。
- VISION_ANALYSIS 输出图片观察。
- TEXT_DRAFT 输出结构化分析。
- QUALITY_CHECK 输出质量分和问题。
- 如果初稿不通过，TEXT_REVISION 会执行。
- 如果最终通过，页面展示完整结果。

### TC-103 质量检查触发自动修订

测试步骤：

1. 使用容易产生复杂功能描述的商品，例如智能音箱类商品。
2. 等待分析完成或失败。
3. 展开 QUALITY_CHECK 和 TEXT_REVISION。

预期结果：

- 如果初稿包含无依据卖点，QUALITY_CHECK 应显示 `UNSUPPORTED_CLAIM`。
- TEXT_REVISION 应显示修订输入和输出。
- 如果模型修订仍不合格，系统应尝试确定性兜底。
- 最终卖点应尽量回到有证据支持的内容。

## 7. API 配置测试

前置条件：

```text
APP_MODE=live
RAINFOREST_API_KEY=
DASHSCOPE_API_KEY=有效值
DEEPSEEK_API_KEY=有效值
```

测试步骤：

1. 重启后端。
2. 观察后端启动日志。

预期结果：

- 后端启动失败或报告缺少环境变量。
- 错误中不泄漏真实 key。

### TC-302 Live 模式缺少 DashScope key

预期结果：

- 图片分析无法调用。
- 错误应定位到图片分析或服务配置。

### TC-303 Live 模式缺少 DeepSeek key

预期结果：

- 文本分析无法调用。
- 错误应定位到文本模型配置。

### TC-304 API key 错误

测试步骤：

1. 将某个 key 改为明显错误值。
2. 重启后端。
3. 提交商品链接。

预期结果：

- 对应阶段失败。
- trace 中当前阶段 failed，后续阶段 skipped。
- 错误信息不显示完整 key。

## 8. 商品数据边界测试

### TC-401 Rainforest 返回 ASIN 不一致

测试目标：

- 验证系统不会接受错误商品数据。

预期结果：

- 商品抓取阶段失败。
- 错误信息说明返回 ASIN 与请求不一致。

### TC-402 商品缺少图片

测试目标：

- 验证图片分析边界。

预期结果：

- VISION_ANALYSIS 失败。
- TEXT_DRAFT、QUALITY_CHECK、TEXT_REVISION、FINALIZE 跳过。

### TC-403 商品缺少 features

测试目标：

- 验证系统能否使用 specifications、evidence_texts 和图片观察支撑卖点。

预期结果：

- 如果 evidence_texts 充足，分析可以继续。
- 如果证据不足，质量检查应标记无依据卖点。

### TC-404 商品缺少 specifications

预期结果：

- 系统不应崩溃。
- specifications 显示为空对象。
- 分析继续使用其他证据。

### TC-405 商品图片很多

测试目标：

- 验证图片数量限制。

预期结果：

- 后端最多传入前 6 张图片给视觉模型。
- trace input 中 image_count 不超过 6。

## 9. 模型输出边界测试

### TC-501 DeepSeek 返回非法 JSON

测试目标：

- 验证结构化解析失败处理。

预期结果：

- TEXT_DRAFT 或 TEXT_REVISION 失败。
- 后续阶段 skipped。
- 前端显示分析失败。

### TC-502 DeepSeek 返回空字段

测试目标：

- 验证完整性检查。

预期结果：

- 系统识别分析内容为空或不完整。
- 不进入最终成功状态。

### TC-503 DeepSeek 修订只返回部分字段

测试目标：

- 验证系统能合并部分修订结果。

预期结果：

- 缺失字段使用原分析字段补齐。
- 修改字段使用新返回值。
- 如果最终质量通过，任务成功。

### TC-504 DeepSeek 修订后仍有无依据卖点

测试目标：

- 验证确定性兜底。

预期结果：

- 系统删除无依据卖点。
- 从 evidence_texts、features、specifications 或 visual_findings 生成可支持卖点。
- 质量检查再次执行。

### TC-505 口播超过 150 字

预期结果：

- QUALITY_CHECK 出现 `SCRIPT_TOO_LONG`。
- 自动修订应尝试缩短口播。

### TC-506 口播没有开头钩子

预期结果：

- QUALITY_CHECK 出现 `WEAK_HOOK`。
- 自动修订应尝试用问题、痛点或反差开头。

## 10. 任务和重试测试

### TC-601 创建任务后轮询状态

测试步骤：

1. 提交商品链接。
2. 使用浏览器观察页面状态。
3. 或调用：

```bash
curl -i http://127.0.0.1:8000/api/jobs/{job_id}
```

预期结果：

- job 状态从 queued/running 变为 completed 或 failed。
- progress 逐步增加。
- trace_events 始终存在 6 个阶段。

### TC-602 失败任务重试

测试步骤：

1. 制造一次失败，例如临时配置错误 API key。
2. 页面显示失败后点击重试。

预期结果：

- job 状态重置为 queued。
- progress 重置为 0。
- trace_events 重置。
- 系统重新执行分析。

### TC-603 运行中任务重复重试

测试目标：

- 验证并发保护。

预期结果：

- 如果任务仍在 queued 或 running，后端返回 409。
- 前端不应启动重复任务。

## 12. 部署测试

### TC-801 Nginx 配置检查

命令：

```bash
sudo nginx -t
```

预期结果：

```text
syntax is ok
test is successful
```

### TC-802 Nginx 首页访问

命令：

```bash
curl -I http://服务器公网IP
```

预期结果：

```text
HTTP/1.1 200 OK
```

如果出现 500，检查：

```bash
sudo tail -n 80 /var/log/nginx/error.log
```

常见原因：

- root 指向 `/root/product-lens-ai/frontend/dist`，Nginx 没权限。
- 应改为 `/opt/product-lens-ai/frontend/dist`。

### TC-803 后端服务状态

命令：

```bash
sudo systemctl status product-lens-backend --no-pager
```

预期结果：

- 状态为 active running。

如果失败，查看：

```bash
sudo journalctl -u product-lens-backend -n 80 --no-pager
```

### TC-804 后端健康检查

命令：

```bash
curl -i http://127.0.0.1:8000/api/health
```

预期结果：

- 返回 HTTP 200。
- 返回 mode。

### TC-805 Nginx API 代理

命令：

```bash
curl -i http://服务器公网IP/api/health
```

预期结果：

- 返回 HTTP 200。
- 返回 JSON。

如果本机 8000 可访问但公网 `/api/health` 不可访问，说明 Nginx proxy 配置有问题。

## 13. 安全测试

### TC-901 前端不泄露 API key

测试步骤：

1. 打开浏览器开发者工具。
2. 查看 Network 返回内容。
3. 查看页面源码。

预期结果：

- 不出现 `RAINFOREST_API_KEY`。
- 不出现 `DASHSCOPE_API_KEY`。
- 不出现 `DEEPSEEK_API_KEY`。
- 不出现 `Authorization` token。

### TC-902 trace 不泄露密钥

测试步骤：

1. 提交任务。
2. 展开所有分析过程。
3. 搜索 `api_key`、`authorization`、`Bearer`。

预期结果：

- 不应出现敏感字段。

### TC-903 后端不暴露 8000 公网端口

测试步骤：

1. 检查阿里云安全组。
2. 尝试访问 `http://服务器公网IP:8000/api/health`。

预期结果：

- 安全组不开放 8000。
- 外部不能直接访问后端。
- 只能通过 Nginx `/api` 访问。

## 14. 极端情况测试

### TC-1001 连续提交多个任务

测试步骤：

1. 快速连续提交多个不同商品链接。
2. 观察任务状态。

预期结果：

- 页面以最后一次提交任务为当前展示。
- 后端不应崩溃。

### TC-1002 外部 API 超时

测试目标：

- 验证超时和错误展示。

预期结果：

- 对应阶段失败。
- 页面显示失败。
- 后续阶段 skipped。

### TC-1003 商品证据极少

测试步骤：

1. 使用信息很少的商品链接。

预期结果：

- 如果证据不足，质量检查不应放行大量推测卖点。
- 最终可能失败，也可能只输出很少的有证据卖点。

### TC-1004 高复杂商品

测试步骤：

1. 使用智能设备、电子产品、多功能工具等商品。

预期结果：

- 系统应尽量从 evidence_texts 支撑卖点。
- 无法支撑的功能应被标记或删除。

### TC-1005 页面刷新

测试步骤：

1. 分析完成后刷新页面。

预期结果：

- 当前版本任务状态不持久化到浏览器或数据库，刷新后页面可能丢失当前展示。
- 这是当前版本限制，不视为缺陷。

## 15. 测试通过标准

本轮手动测试通过标准：

- 首页可访问。
- `/api/health` 正常。
- Demo 模式完整流程成功。
- Live 模式至少一个真实商品链接成功。
- 非 Amazon 链接被拒绝。
- 失败任务能显示原因并支持重试。
- 分析过程 6 个阶段均可查看。
- 无依据卖点能被质量检查发现。
- API key 不在前端和 trace 中泄露。
- 云服务器 Nginx 和 systemd 服务均正常。
