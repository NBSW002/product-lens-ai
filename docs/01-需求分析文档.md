# ProductLens 需求分析文档

版本：1.0  
日期：2026-07-16  
项目名称：ProductLens AI 商品内容分析系统

## 1. 项目背景

跨境电商商品内容分析通常需要人工查看商品详情页、主图、卖点、规格、价格、评分和用户场景，再整理成适合短视频或内容平台使用的文案。这个过程存在几个问题：

- 人工分析耗时，且不同人员输出风格不稳定。
- 商品卖点容易凭经验发挥，缺少证据来源。
- 图片信息和商品结构化字段经常割裂，无法形成完整内容判断。
- AI 生成内容容易出现无依据卖点、夸大表达、口播过长等问题。
- 用户只看到最终结果时，无法追溯每一步输入、输出和错误原因。

ProductLens 的目标是把一个公开 Amazon 商品链接转化为“有证据依据的商品洞察和短视频口播文案”，同时把商品抓取、图片分析、AI 初稿、质量检查、自动修订和最终结果透明展示出来。

## 2. 目标用户

主要用户：

- 跨境电商运营人员
- 小红书、TikTok、短视频内容策划人员
- Amazon 商品分析人员
- 广告素材策划和选品人员

次要用户：

- 电商 SaaS 产品演示人员
- AI 内容生成工具评估人员
- 需要快速理解商品卖点和场景的业务人员

## 3. 用户痛点

1. 商品信息分散  
   Amazon 商品页包含标题、价格、评分、规格、图片、描述、A+ 内容等，人工整理成本高。

2. AI 输出不可控  
   大模型可能补充页面没有提供的能力、规格或使用场景，导致内容不可直接使用。

3. 分析过程不透明  
   如果只显示最终文案，用户无法判断结论来自商品事实、图片观察还是模型推断。

4. 失败原因不清楚  
   外部 API 失败、图片分析失败、DeepSeek 返回不完整、质量检查不通过等问题需要被清晰定位。

5. 内容需要可复用  
   用户希望直接获得目标用户、使用场景、痛点、卖点、图片观察和短视频口播，不只是普通摘要。

## 4. 项目目标

系统需要实现以下目标：

- 输入 Amazon 商品链接，自动识别 ASIN 和站点域名。
- 获取商品事实数据，包括标题、分类、价格、评分、评论数、卖点、规格、图片和额外证据文本。
- 使用视觉模型分析商品图片，提取可见且与购买决策相关的信息。
- 使用文本模型生成结构化产品分析。
- 对 AI 输出进行质量检查，发现无依据卖点、夸大表达、口播过长、弱开头等问题。
- 在质量检查不通过时自动修订内容。
- 如果模型修订后仍保留无依据卖点，系统使用确定性兜底逻辑删除或替换为有证据支持的卖点。
- 前端展示完整流水线，使用户可以展开查看每一步输入、输出、耗时、错误和字段来源。
- 支持 Demo 模式和真实 API 模式。
- 支持云服务器部署。

## 5. 功能需求

### 5.1 Amazon 链接输入

用户可以在页面输入 Amazon 商品链接。系统需要：

- 校验链接是否属于 Amazon 域名。
- 从链接中提取 ASIN。
- 标准化为 canonical URL。
- 拒绝非 Amazon 链接、缺少 ASIN 的链接和明显非法 URL。

### 5.2 商品数据抓取

真实 API 模式下，系统通过 Rainforest API 获取商品数据。需要抓取或整理：

- ASIN
- 标题
- 分类
- 价格
- 货币
- 评分
- 评论数量
- feature bullets
- specifications
- 商品图片
- description、feature sections、A+ content、product overview、technical details 等额外证据文本

Demo 模式下，系统使用固定数据返回可复现结果。

### 5.3 图片分析

系统需要将商品图片传入 Qwen Vision 模型，输出图片观察结果。图片分析应满足：

- 只描述图片中可见内容。
- 不猜测不可见功能。
- 关注结构、材质、配件、尺寸标注和使用场景。
- 输出结构化 JSON。
- 如果没有图片或模型返回空观察，应明确失败。

### 5.4 文本分析

系统通过 DeepSeek 生成结构化商品分析，输出字段包括：

- target_users：目标用户
- scenarios：使用场景
- pain_points：用户痛点
- selling_points：有证据支持的核心卖点
- visual_findings：图片观察
- voiceover：150 字以内中文短视频口播

文本模型必须基于商品事实和图片观察，不应编造页面未提供的信息。

### 5.5 质量检查

系统需要对 AI 初稿和修订稿执行质量检查，检查项包括：

- 必填字段是否为空。
- 卖点是否能在商品事实、规格、额外证据文本或图片观察中找到依据。
- 是否包含绝对化或夸大词。
- 口播是否超过 150 字。
- 口播开头是否有问题、痛点或反差钩子。

质量检查输出：

- score：质量分数
- passed：是否通过
- evidence_coverage：卖点证据覆盖率
- issues：问题列表，包含 code、severity、message、suggestion

### 5.6 自动修订

当质量检查不通过时，系统需要：

- 把质量问题传给 DeepSeek。
- 要求模型只修订问题内容。
- 允许模型返回部分字段，系统自动用原分析补齐缺失字段。
- 再次执行质量检查。

如果修订后仍存在无依据卖点，系统应执行确定性兜底：

- 删除无依据卖点。
- 从商品 features、specifications、evidence_texts、visual_findings 中生成有证据支持的替代卖点。
- 再次执行质量检查。

### 5.7 透明流水线

前端需要展示每个阶段的执行状态：

- PRODUCT_FETCH：商品抓取
- VISION_ANALYSIS：图片分析
- TEXT_DRAFT：DeepSeek 初稿
- QUALITY_CHECK：质量检查
- TEXT_REVISION：自动修订
- FINALIZE：最终结果

每一步支持展示：

- 状态：pending、running、completed、failed、skipped
- provider 和 model
- 开始时间、结束时间、耗时
- 输入
- 输出
- 字段来源
- 错误信息

### 5.8 任务重试

当任务失败时，用户可以点击重试。系统需要：

- 保留原始 URL。
- 重置任务状态和 trace events。
- 重新执行完整分析流程。
- 避免对正在运行的任务重复启动。

## 6. 非功能需求

### 6.1 可用性

- 前端页面应能直接展示进度和结果。
- 失败时要给出可读错误。
- Demo 模式可在没有 API key 的情况下完整演示。

### 6.2 可追溯性

- 每个核心阶段都要有 trace event。
- 商品字段要标注来源。
- 质量问题要可展开查看。

### 6.3 安全性

- API key 只能保存在后端 `.env` 中。
- 前端不得暴露任何密钥。
- 后端应拒绝非 Amazon URL。
- 错误信息需要经过清理，避免泄露 Authorization、api_key 等敏感字段。

### 6.4 可部署性

- 前端可构建为静态文件并由 Nginx 托管。
- 后端可通过 systemd 常驻运行。
- Nginx 将 `/api` 反向代理到本机 FastAPI 服务。

### 6.5 可测试性

- 后端 provider、URL 解析、质量检查、服务流程需要自动化测试。
- 前端主要页面和分析流水线需要组件测试。
- 真实 API 可通过手动测试验证。

## 7. 边界与限制

- 当前任务数据使用内存仓库，服务器重启后任务记录会丢失。
- 外部 API 的返回字段受 Rainforest、DashScope、DeepSeek 服务稳定性影响。
- 图片分析只应使用图片可见内容，不保证理解所有商品功能。
- 证据检查采用字符串匹配和规则判断，不能替代人工合规审查。
- 系统当前没有用户登录、权限隔离和费用控制。

## 8. 验收标准

系统达到以下条件可认为当前版本满足需求：

- 输入有效 Amazon 链接后能生成商品概览、洞察、短视频口播和质量评分。
- 无 API key 时 Demo 模式可完整运行。
- Live 模式在配置三个 API key 后可以调用真实服务。
- 非 Amazon 链接会被拒绝。
- 分析过程可展开查看每一步输入、输出和错误。
- 无依据卖点会被质量检查标记，并触发修订或兜底。
- 前端测试、后端测试和前端构建通过。
