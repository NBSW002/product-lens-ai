# ProductLens 架构设计文档

版本：1.0  
日期：2026-07-16  
项目名称：ProductLens AI 商品内容分析系统

## 1. 架构目标

系统采用前后端分离架构，目标是：

- 前端专注交互、进度展示和结果呈现。
- 后端负责链接校验、任务编排、外部 API 调用、质量检查和结果封装。
- 外部服务通过 provider 适配器接入，方便 Demo 模式和真实 API 模式切换。
- 分析流程按阶段记录 trace，便于用户追溯和排错。
- 云服务器部署时通过 Nginx 提供静态页面和 API 反向代理。

## 2. 总体架构

```text
用户浏览器
  |
  | HTTP
  v
Nginx
  |-- 静态文件：frontend/dist
  |
  | /api/*
  v
FastAPI 后端
  |
  | 任务仓库 JobRepository
  | 分析编排 AnalysisService
  |
  |-- RainforestProductProvider  -> Rainforest API
  |-- QwenVisionProvider         -> 阿里云百炼 / Qwen Vision
  |-- DeepSeekTextModel          -> DeepSeek API
  |
  v
质量检查 QualityChecker
```

```text
frontend/src/App.tsx

frontend/src/types.ts
frontend/src/components/
```

### 3.1 前端职责

前端负责：

- 输入 Amazon 商品链接。
- 创建分析任务。
- 轮询任务状态。
- 展示任务进度。
- 展示分析流水线。
- 展示商品摘要、洞察卡片、口播文案、质量报告。
- 在任务失败时提供重试入口。

### 3.2 前端数据流

```text
用户提交 URL
  -> createJob(url)
  -> POST /api/jobs
  -> 后端返回 Job
  -> 前端保存 job
  -> 每 700ms 调用 GET /api/jobs/{job_id}
  -> job.completed 后展示 result
  -> job.failed 后展示错误和重试按钮
```

### 3.3 前端核心组件

- `ProgressTimeline`：显示当前阶段和进度。
- `AnalysisTrace`：展示完整流水线。
- `TraceStep`：展示单个阶段的输入、输出、耗时和错误。
- `ProductSummary`：展示商品基础信息。
- `InsightGrid`：展示目标用户、使用场景、痛点、卖点。
- `ScriptPanel`：展示短视频口播。
- `QualityPanel`：展示质量分、证据覆盖率和质量问题。

### 4.1 API 层

`backend/app/main.py` 提供 API：

| 方法 | 路径 | 说明 |
|---|---|---|
| GET | `/api/health` | 返回服务状态和当前模式 |
| POST | `/api/jobs` | 创建分析任务 |
| GET | `/api/jobs/{job_id}` | 查询任务状态 |
| POST | `/api/jobs/{job_id}/retry` | 重试失败或已完成任务 |

API 层负责：

- 接收请求。
- URL 基础校验。
- 创建任务。
- 添加后台任务。
- 查询任务。
- 将异常转成安全错误信息。

### 4.2 任务仓库

`JobRepository` 当前采用内存存储，负责：

- 创建 job。
- 更新 job 状态、阶段、进度、结果和错误。
- 初始化 trace events。
- 更新每个 trace event 的状态、输入、输出、字段来源和错误。

当前内存仓库适合 Demo 和轻量部署。生产化可以替换为 SQLite、PostgreSQL 或 Redis。

### 4.3 分析编排层

`AnalysisService` 是核心业务编排器，按顺序执行：

1. 校验并解析 Amazon URL。
2. 商品抓取。
3. 图片分析。
4. 文本初稿生成。
5. 第一次质量检查。
6. 自动修订。
7. 第二次质量检查。
8. 必要时确定性兜底修复无依据卖点。
9. 最终结果封装。

`AnalysisService` 通过 Protocol 依赖抽象接口：

```text
ProductProvider.fetch(link) -> ProductFacts
VisionProvider.analyze(images) -> list[str]
TextModel.analyze(facts, visual_findings) -> ProductAnalysis
TextModel.revise(facts, analysis, issue_messages) -> ProductAnalysis
```

这样 Demo provider 和 Live provider 可以复用同一条业务流程。

## 5. Provider 设计

### 5.1 Demo Provider

Demo 模式不依赖外部 API，固定返回商品事实、图片观察和分析结果。用途：

- 无 API key 演示。
- 自动化测试。
- 本地开发。
- 避免消耗真实 API 额度。

### 5.2 RainforestProductProvider

职责：

- 调用 Rainforest API。
- 传入 `api_key`、`amazon_domain`、`asin`、`type=product`。
- 校验返回 ASIN 是否与请求一致。
- 解析商品字段为 `ProductFacts`。

核心输出字段：

- title
- category
- price
- currency
- rating
- review_count
- features
- specifications
- evidence_texts
- images
- source_url

`evidence_texts` 会从 description、feature sections、A+ content、product overview、technical details 等字段递归抽取文本，用于后续证据检查。

### 5.3 QwenVisionProvider

职责：

- 调用阿里云百炼 OpenAI-compatible chat completions 接口。
- 使用 `qwen3-vl-plus`。
- 输入最多 6 张商品图片。
- 要求模型只描述图片中可见内容。
- 输出 JSON：`{"findings": ["..."]}`。

如果没有图片或返回 findings 为空，视为图片分析失败。

### 5.4 DeepSeekTextModel

职责：

- 调用 DeepSeek chat completions 接口。
- 使用 `deepseek-v4-flash`。
- 生成结构化商品分析。
- 修订不通过质量检查的分析结果。

设计要点：

- 要求 JSON object 输出。
- `_request_json` 负责请求和解析 JSON。
- `_validate_analysis` 负责校验完整 `ProductAnalysis`。
- 初稿必须完整返回所有字段。
- 修订阶段允许模型只返回部分字段，系统用原分析补齐缺失字段。
- 显式返回空数组或空口播仍视为无效。

## 6. 质量检查设计

`QualityChecker` 负责确定性质量门禁。

### 6.1 证据池

证据来源包括：

- 商品标题
- 商品分类
- feature bullets
- specification keys
- specification values
- evidence_texts
- 图片观察 visual_findings

### 6.2 检查项

- 必填字段为空。
- 卖点无法在证据池中命中。
- 口播或卖点包含绝对化/夸大词。
- 口播超过 150 字。
- 口播开头缺少问题、痛点或钩子。

### 6.3 评分规则

系统以 100 分为基础，根据问题严重程度扣分：

- low：扣 5 分
- medium：扣 12 分
- high：扣 24 分

通过条件：

- 分数不低于 80。
- 证据覆盖率大于 0。
- 不存在 high 级别问题。

### 6.4 确定性兜底

如果 DeepSeek 修订后仍保留无依据卖点，系统执行：

- 删除 unsupported selling points。
- 如果没有剩余卖点，从 features、specifications、evidence_texts、visual_findings 生成兜底卖点。
- 再次执行质量检查。

该设计避免完全依赖模型自我修订。

## 7. Trace 设计

系统预置 6 个阶段：

| 阶段 | 说明 |
|---|---|
| PRODUCT_FETCH | 商品抓取 |
| VISION_ANALYSIS | 图片分析 |
| TEXT_DRAFT | 文本初稿 |
| QUALITY_CHECK | 质量检查 |
| TEXT_REVISION | 自动修订 |
| FINALIZE | 最终确认 |

每个 trace event 包含：

- id
- stage
- title
- status
- provider
- model
- started_at
- finished_at
- duration_ms
- input
- output
- field_sources
- error

失败时，当前阶段标记为 failed，后续未执行阶段标记为 skipped。

## 8. 部署架构

推荐部署方式：

```text
阿里云 ECS
  |
  | Nginx :80/:443
  |-- /                 -> /opt/product-lens-ai/frontend/dist
  |-- /api/*            -> http://127.0.0.1:8000/api/*
  |
  | systemd
  |-- product-lens-backend.service
      -> uvicorn app.main:app --host 127.0.0.1 --port 8000
```

环境变量位于：

```text
/opt/product-lens-ai/backend/.env
```

关键变量：

```text
APP_MODE=live
RAINFOREST_API_KEY=...
DASHSCOPE_API_KEY=...
DEEPSEEK_API_KEY=...
```

## 9. 安全设计

- API key 只在后端 `.env` 中保存。
- 前端不接触任何 API key。
- `/api/jobs` 会校验 URL，只允许 Amazon 商品链接。
- 错误信息经过 `sanitize_error` 处理，避免泄漏密钥。
- 后端服务只监听 `127.0.0.1:8000`，不直接暴露公网。
- Nginx 作为唯一公网入口。

## 10. 当前架构限制

- JobRepository 是内存实现，重启丢数据。
- 没有用户账号和权限控制。
- 没有 API 调用成本统计。
- 没有后台队列，任务在 FastAPI background task 中执行。
- 质量检查以规则和字符串匹配为主，复杂语义证据匹配仍有限。
