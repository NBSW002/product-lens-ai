# ProductLens 核心功能与实现说明

版本：1.0  
日期：2026-07-16  
项目名称：ProductLens AI 商品内容分析系统

## 1. 系统简介

ProductLens 是一个面向 Amazon 商品链接的 AI 产品分析工具。用户输入商品链接后，系统会自动抓取商品数据、分析商品图片、生成产品洞察和短视频口播，并通过质量检查保证内容尽量基于可追溯证据。

系统当前支持两种模式：

- Demo 模式：不需要 API key，使用固定数据演示完整流程。
- Live 模式：使用 Rainforest API、阿里云 Qwen Vision、DeepSeek API 生成真实分析结果。

## 2. 核心功能一：Amazon 链接解析与校验

### 功能说明

用户输入 Amazon 商品链接后，系统需要识别该链接是否有效，并提取 ASIN。

### 实现方式

后端使用 `backend/app/url_parser.py` 处理 URL：

- 校验域名是否属于 Amazon。
- 支持常见 Amazon 商品 URL 格式。
- 从路径中提取 ASIN。
- 输出 `AmazonLink`，包含 ASIN、marketplace host、canonical URL。

### 异常处理

如果 URL 非 Amazon 域名、缺少 ASIN 或格式非法，接口返回 422，前端展示错误信息。

## 3. 核心功能二：商品数据抓取

### 功能说明

系统需要获得商品事实，作为后续分析和证据检查的基础。

### Demo 实现

Demo 模式使用 `DemoProductProvider` 返回固定商品数据，不依赖外部服务。

### Live 实现

Live 模式使用 `RainforestProductProvider` 调用 Rainforest API：

```text
GET https://api.rainforestapi.com/request
```

请求参数包括：

- api_key
- type=product
- amazon_domain
- asin

### 输出字段

抓取结果统一转换为 `ProductFacts`：

- asin
- title
- category
- price
- currency
- rating
- review_count
- features
- specifications
- evidence_texts
- images
- source_url

### 额外证据文本

为了减少 AI 无依据发挥，系统新增了 `evidence_texts` 字段。它从 Rainforest 返回的以下字段中递归抽取文本：

- description
- feature_sections
- a_plus_content
- product_overview
- technical_details
- important_information
- product_description
- detail_bullets

这些文本不会直接作为最终文案，而是作为质量检查的证据池。

## 4. 核心功能三：图片分析

### 功能说明

商品图片通常包含结构、配件、材质、尺寸标注、使用场景等信息。系统会将商品图片交给视觉模型分析。

### 实现方式

Live 模式使用 `QwenVisionProvider`：

- 接口：阿里云 DashScope OpenAI-compatible chat completions
- 模型：`qwen3-vl-plus`
- 输入：最多 6 张商品图片
- 输出：JSON 格式 findings

提示词要求模型：

- 只描述图片中可见内容。
- 不猜测不可见功能。
- 关注购买决策相关信息。

### 失败条件

- 商品没有图片。
- 视觉模型返回空 findings。
- API 请求失败。
- JSON 无法解析。

这些情况会导致 `VISION_ANALYSIS` 阶段失败，后续阶段跳过。

## 5. 核心功能四：DeepSeek 商品分析

### 功能说明

系统将商品事实和图片观察传入 DeepSeek，生成结构化产品分析。

### 输出结构

`ProductAnalysis` 包含：

- target_users：目标用户
- scenarios：使用场景
- pain_points：用户痛点
- selling_points：核心卖点
- visual_findings：图片观察
- voiceover：150 字以内中文口播

### 实现方式

`DeepSeekTextModel.analyze()` 构造提示词，要求模型：

- 只使用输入证据。
- 不虚构。
- 输出合法 JSON。
- 区分事实与推断。
- 口播避免绝对化宣传。

### 结构校验

后端会用 Pydantic 校验模型输出。如果字段缺失、字段类型错误或核心字段为空，则抛出安全错误。

## 6. 核心功能五：质量检查

### 功能说明

AI 初稿不能直接视为可用内容，必须经过规则化质量检查。

### 实现文件

```text
backend/app/quality.py
```

### 检查内容

1. 完整性检查  
   检查 target_users、scenarios、pain_points、selling_points、visual_findings、voiceover 是否为空。

2. 证据检查  
   每个 selling point 必须能在证据池中找到依据。

3. 夸大表达检查  
   检查“全球第一”“百分百”“100%”“绝对”“最强”等绝对化表达。

4. 口播长度检查  
   口播去空白后不应超过 150 字。

5. 开头钩子检查  
   口播前 28 个字符中应包含问题、痛点或反差表达。

### 证据池组成

```text
title
category
features
specifications.keys
specifications.values
evidence_texts
visual_findings
```

### 输出结果

质量检查输出 `QualityReport`：

- score
- passed
- evidence_coverage
- issues

每个 issue 包含：

- code
- severity
- message
- suggestion

## 7. 核心功能六：自动修订

### 功能说明

当质量检查不通过时，系统自动把问题传给 DeepSeek，让模型修订内容。

### 实现方式

`DeepSeekTextModel.revise()` 输入：

- 商品事实
- 原始分析
- 质量问题列表

输出仍为 `ProductAnalysis`。

### 部分 JSON 兼容

实际使用中，DeepSeek 修订时可能只返回修改过的字段。系统已经处理这个情况：

- 如果修订结果缺少字段，使用原分析字段补齐。
- 如果修订结果显式返回空数组或空字符串，仍判为无效。

这样可以避免因为模型只返回部分 JSON 导致整个任务失败。

## 8. 核心功能七：确定性兜底修复

### 功能说明

如果 DeepSeek 修订后仍保留无依据卖点，系统不能继续完全依赖模型。此时后端执行确定性兜底。

### 实现方式

`QualityChecker.repair_unsupported_claims()`：

1. 遍历当前 selling_points。
2. 保留能在证据池中命中的卖点。
3. 如果没有任何可保留卖点，从以下来源生成兜底卖点：
   - features
   - specifications
   - evidence_texts
   - visual_findings
4. 再次执行质量检查。

### 价值

该机制解决的问题是：模型知道要修，但仍然保留了页面没有证据支持的说法。兜底逻辑可以把输出拉回“可验证证据”范围。

## 9. 核心功能八：透明分析流水线

### 功能说明

用户不仅需要最终结果，还需要看到每一步如何产生。系统通过 trace events 记录完整过程。

### Trace 阶段

```text
PRODUCT_FETCH
VISION_ANALYSIS
TEXT_DRAFT
QUALITY_CHECK
TEXT_REVISION
FINALIZE
```

### 每个阶段记录

- 输入
- 输出
- 状态
- provider
- model
- 字段来源
- 开始时间
- 结束时间
- 耗时
- 错误

### 前端展示

`AnalysisTrace` 和 `TraceStep` 负责把这些信息展示为可展开流水线。用户可以查看商品抓取内容、图片分析结果、DeepSeek 初稿、质量检查问题、修订结果和最终状态。

## 10. 核心功能九：任务状态和重试

### 任务状态

Job 包含：

- id
- url
- status
- stage
- progress
- result
- error
- trace_events
- created_at
- updated_at

### 状态流转

```text
queued -> running -> completed
queued -> running -> failed
failed/completed -> retry -> queued
```

### 重试逻辑

用户点击重试时：

- 原 URL 保留。
- 结果、错误、进度清空。
- trace events 重置。
- 后台重新执行完整流程。

如果任务仍在 running 或 queued，系统拒绝重复启动。

## 11. 核心功能十：部署

### 推荐部署方式

- 前端执行 `npm run build`，生成 `frontend/dist`。
- Nginx 托管 `frontend/dist`。
- FastAPI 通过 systemd 运行在 `127.0.0.1:8000`。
- Nginx 将 `/api/` 转发到后端。

### 推荐路径

```text
/opt/product-lens-ai
```

不建议放在 `/root/product-lens-ai`，因为 Nginx 的 `www-data` 用户通常无权读取 `/root` 目录。

### systemd 服务

```ini
WorkingDirectory=/opt/product-lens-ai/backend
EnvironmentFile=/opt/product-lens-ai/backend/.env
ExecStart=/opt/product-lens-ai/backend/.venv/bin/python -m uvicorn app.main:app --host 127.0.0.1 --port 8000
```

## 12. 当前实现中的重要取舍

1. 内存任务仓库  
   部署简单，但重启丢任务。

2. FastAPI BackgroundTasks  
   实现轻量，但不适合大量并发任务。

3. 规则质量检查  
   可解释、稳定，但语义匹配能力有限。

4. Demo/Live 同流程  
   便于测试和演示，但 Demo 数据不能代表真实 API 的所有边界。
